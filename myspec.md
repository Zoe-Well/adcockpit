# SPEC.md: AI 数字营销优化师 —— 对话驾驶舱

> **项目代号**：AdCockpit  
> **目标岗位**：数字营销 AI Agent System 架构师  
> **核心表现形式**：对话驾驶舱（Conversational Cockpit） = 左侧对话面板 + 中间 Agent 编排与执行追踪 + 右侧数据洞察仪表盘

---

## 1. 项目概述与业务意义

### 1.1 业务背景
作为抖音、腾讯视频号、快手、小红书等头部平台的官方授权服务商，我们为 B 端企业客户提供**广告投放、内容生产、电商闭环、数据分析**的全链路数字营销服务。

核心痛点：
- **多平台数据孤岛**：巨量引擎、磁力引擎、腾讯广告、小红书聚光各自独立，人工汇总分析耗时易错。
- **决策-执行链路断裂**：从发现问题到跨平台执行优化，中间需要多工具切换，响应滞后。
- **内容-投放-电商割裂**：内容生产、广告投放、直播间/店铺运营往往由不同团队负责，缺乏统一调度的“AI 中枢”。
- **客户信任与透明度缺失**：客户把预算和生意交给代理公司，必须能看懂 AI 在干什么、为什么这么干，才能放心放权。

### 1.2 Demo 目标
构建名为 **AdCockpit** 的 AI Agent 系统，以 **“对话驾驶舱”** 的混合 UI 形式，演示如何通过自然语言指令驱动多个 Agent 协作，自动完成以下四类核心任务：
1. **广告投放优化**：跨平台数据分析、异常检测、自动调价/调预算
2. **内容生产与分发**：基于投后数据的素材分析、AI 混剪/改稿、多平台一键发布
3. **电商闭环运营**：直播间实时监控、自动优惠券/库存调整、客服话术辅助
4. **数据分析与洞察**：自定义报表生成、归因分析、预算分配建议

整个系统将在统一的驾驶舱界面中透明展示 Agent 的“思考”与“行动”，支持人工干预，让客户和运营团队真正信赖 AI。

## 2. 核心用户场景 (User Stories)

### 场景一：跨平台广告投放智能优化官
**对应领域**：广告投放  
**用户角色**：优化师小王  
**场景描述**：  
小王早上到岗，在左侧对话区输入：“帮我检查一下最近 7 天在抖音和腾讯广告上，消耗排名 Top 5 的计划里哪些 ROI 低于 2，对不达标的计划自动降价 10%，然后生成一份今日优化报告。”

**Agent 编排与 UI 展示**：
- **调度器 Agent** 拆解任务：拉取抖音数据、拉取腾讯数据、合并分析、生成策略、执行调价、输出报告。
- **数据 Agent** 并行调用 `get_top_campaigns(platform="douyin", days=7)` 和 `get_top_campaigns(platform="tencent", days=7)`，中间看板展示两步同时进行。
- **分析 Agent** 筛选出 `roi < 2` 的计划，右侧仪表盘高亮异常卡片（红框）。
- **策略 Agent** 生成动作：计划 A 降价 10%，计划 B 降价 10% 且降预算 20%。
- **人工审批中断**：左侧弹出确认卡片“即将对以下 3 条计划执行调价：… 确认执行？”，小王点击确认。
- **执行 Agent** 调用 `update_bid` 等 API，显示结果，右侧自动生成优化报告摘要：预计节省消耗 15%，ROI 提升至 2.1。
- **记忆与后续**：系统记住小王偏好“只关注 ROI<2 的计划”，下次类似指令会自动应用该过滤条件。

---

### 场景二：投后素材分析与 AI 内容工厂
**对应领域**：内容生产  
**用户角色**：内容运营小张  
**场景描述**：  
小张在投放两天后想优化素材，输入：“把昨天抖音上点击率最高和最低的 3 条视频找出来，分析一下高点击视频的共性，然后基于爆款模板帮我生成 3 条新的带货口播脚本，发布到我们的飞书内容库。”

**Agent 编排与 UI 展示**：
- **调度器 Agent** 将任务拆为：数据拉取、素材分析、脚本生成、内容发布。
- **数据 Agent** 调用 `get_top_creatives(platform="douyin", metric="ctr", date="yesterday", top_n=3, worst_n=3)`，右侧展示视频封面、CTR、完播率等。
- **分析 Agent** 调用多模态 LLM 对高/低点击视频进行帧分析，中间卡片显示“发现共性：前 3 秒有价格锚点、口播语速快、BGM 为热门卡点音乐”。
- **内容生产 Agent** 基于分析结果，调用 AI 混剪模板库和 LLM 生成 3 条新口播脚本，右侧内容预览区逐条展示，并附带“一键改改”按钮供人工微调。
- **发布 Agent** 通过工具 `publish_to_feishu(repo="内容库", files=scripts)` 将脚本推送到飞书知识库，中间步骤显示上传成功。
- **长期记忆**：系统记录“爆款模板-夏季促销”，下次可直接调用。

---

### 场景三：直播间实时 AI 场控与电商闭环
**对应领域**：电商闭环  
**用户角色**：直播运营小刘  
**场景描述**：  
小刘正在盯播，在对话框粘贴实时大屏数据：“当前直播间在线 2000 人，商品 A 点击率很高但转化率突然下跌。帮我查一下是不是库存快没了，如果库存低于 50 件就自动追加 200 件，同时创建一个限时 10 元的优惠券发在评论区，并给主播生成一条催单话术。”

**Agent 编排与 UI 展示**：
- **调度器 Agent** 识别为紧急任务，高优先级路由。
- **数据 Agent** 调用 `get_product_stock(product_id="A")`，中间卡片显示“当前库存：32 件”，标红警告。
- **执行 Agent** 并行执行：
    - 调用 `update_stock(product_id="A", add=200)` → 库存更新至 232 件
    - 调用 `create_coupon(product_id="A", discount=10, channel="live_comment")` → 优惠券生成，右侧仪表盘实时展示券领取情况
- **内容生产 Agent** 调用 LLM 生成主播催单话术：“库存已补！现在下单立减 10 元，只剩最后 200 件，手慢无！” 并直接推送到场控中控台或弹窗显示。
- **电商闭环联动**：同时自动将本次补货操作记录到 ERP 模拟日志，并在右侧显示“预计带动 GMV +5000 元”的预估。
- **人工干预点**：如库存增加操作前，若设定了规则“库存变更需确认”，则先弹出确认窗。

### 场景四：跨渠道数据分析与预算决策助手
**对应领域**：数据分析  
**用户角色**：客户总监老李  
**场景描述**：  
老李需要为下周客户提案，输入：“拉取本月截至昨天，我们所有客户在抖音、腾讯、小红书的整体消耗、ROI 和 CPA，按客户分组汇总，找出预算回报率最高的 3 个客户和最低的 2 个客户，并给我一份下周预算再分配建议，做成 PPT 提纲。”

**Agent 编排与 UI 展示**：
- **调度器 Agent** 分解为：多平台数据聚合、客户维度聚合、排名分析、建议生成、格式化输出。
- **数据 Agent** 依次/并行调用各平台 Reporting API（模拟），中间看板逐平台显示拉取进度条。
- **分析 Agent** 对汇总数据按客户分组计算 ROI 和 CPA，右侧仪表盘用柱状图展示“客户 A 消耗最多但 ROI 仅 1.5，客户 C 消耗低但 ROI 3.2”。
- **策略 Agent** 基于规则+LLM 生成预算建议：“将客户 A 预算削减 20% 转移至客户 C，预计整体 ROI 提升 0.3”。
- **报告 Agent** 调用 `generate_ppt_outline` 工具，生成包含数据图表、建议和话术的 PPT 提纲，右侧直接显示可复制的大纲文本。
- **记忆**：保存本次分析维度为“月度客户汇报”，下次可一键复用。

### 场景五：全链路故障诊断与自主恢复
**对应领域**：全链路（运维/稳定性）  
**用户角色**：优化师小王  
**场景描述**：  
小王发现某条计划突然没量，输入：“计划 ID 12345 在抖音上突然没有消耗了，帮我排查一下是什么原因，如果是出价问题就提价 5%，如果是审核被拒就通知我，并把素材替换为备用视频。”

**Agent 编排与 UI 展示**：
- **调度器 Agent** 启动诊断子任务流。
- **数据 Agent** 调用 `get_plan_status(plan_id=12345)`，获取状态、审核状态、出价信息等。
- **分析 Agent** 发现状态为“审核拒绝”，原因：素材涉及虚假承诺。中间卡片标红显示原因，右侧显示被拒素材截图。
- **策略 Agent** 给出决策：通知人工审核，并调用 `replace_creative(plan_id=12345, new_video_id="backup_001")` 自动替换为备用视频。
- **执行 Agent** 发送飞书通知给小王：“计划 12345 因素材审核被拒，已自动替换为备用视频并提交审核”，同时调用 `resubmit_plan(plan_id=12345)`。
- **全链路价值**：从检测到恢复完全自动化，且所有步骤透明可追溯。

## 3. UI 布局与表现形式

Demo 采用 **三栏布局**，适应所有上述场景，只是在右侧仪表盘根据场景自动切换视图。

```
+------------------+---------------------------+------------------------+
| 左侧：对话面板    | 中间：Agent 编排与执行追踪  | 右侧：数据洞察仪表盘      |
| (Chat Panel)     | (Trace Board)              | (Insight Dashboard)     |
|                  |                           |                        |
| - 自然语言输入   | - 任务分解图谱             | - 场景自适应：          |
| - 多轮对话澄清   | - Multi-Agent 执行步骤流   |   * 投放：ROI 卡片/趋势 |
| - 历史记录       | - 工具调用详情 (Req/Res)   |   * 内容：脚本预览/修改 |
| - 确认/干预弹窗  | - 状态标识 (等待/进行中/   |   * 电商：库存/券看板   |
|                  |   成功/失败/需确认)        |   * 数据：图表/PPT 提纲 |
|                  | - 错误重试与回滚           | - 最终报告区           |
+------------------+---------------------------+------------------------+
```

## 4. 技术架构与实现

### 4.1 整体架构
```
前端 (Streamlit/Gradio 三栏布局 + WebSocket)
   ↕
后端 (FastAPI)
   ↕
LangGraph Agent 编排层 (StateGraph)
   ├── Supervisor Agent (任务规划与动态路由)
   ├── Data Agent (多平台数据拉取 Mocks)
   ├── Analysis Agent (数据分析与诊断)
   ├── Strategy Agent (策略生成与预算分配)
   ├── Content Agent (文案/脚本/视频分析)
   ├── Ecommerce Agent (库存/优惠券/直播)
   ├── Execute Agent (统一操作执行 + 人工确认中断)
   └── Report Agent (可视化报告/PPT 提纲)
   ↕
工具层 (30+ Mock 函数覆盖巨量、腾讯、小红书、快手、飞书、ERP 等)
   ↕
LLM (GPT-4o / Claude 3.5 Sonnet, 支持 Function Calling 与多模态)
```

### 4.2 关键设计如何覆盖 JD 要求

| JD 要求 | Demo 体现 |
|---------|-----------|
| **Multi-Agent 编排框架** | LangGraph 实现有状态图，包含 7+ 专业 Agent，支持并行、条件分支、中断与恢复。 |
| **工具调用链路** | 每个工具调用在中间面板完整展示输入输出、耗时和状态，可展开查看详情。 |
| **记忆与状态管理** | 短期记忆通过对话历史，长期记忆存储用户偏好、爆款模板、分析维度；状态持久化支持中断后恢复。 |
| **打通平台 API** | 工具层模拟巨量引擎、磁力引擎、腾讯广告、小红书聚光、飞书、电商 ERP 等 API，Schema 设计贴近真实。 |
| **业务理解** | 场景覆盖广告投放、内容生产、电商闭环、数据分析全链路，每个操作均基于真实业务指标和动作。 |
| **AI Coding 工作流** | Demo 代码 95% 由 Claude Code/Codex 生成，开发过程完全遵循 AI Coding 范式。 |
| **OpenClaw/LangChain/CrewAI** | 首选 LangGraph，同时可在面试中展示如何迁移至 OpenClaw 的可视化编排环境。 |

## 5. 开发与面试准备
