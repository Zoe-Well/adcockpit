# AdCockpit — AI 数字营销对话驾驶舱 · 项目完整介绍

> **一句话定位**：基于 LangGraph Multi-Agent 编排的全链路数字营销 AI 系统，
> 以"对话驾驶舱"的三栏 UI 形态，通过自然语言驱动 8 个专业 Agent 协作，
> 覆盖广告投放、内容生产、电商闭环、数据分析四大核心业务。

---

## 一、项目背景与业务价值

### 1.1 行业痛点

作为抖音、腾讯视频号、快手、小红书等头部平台的官方授权服务商，
我们为 B 端企业客户提供全链路数字营销服务。当前行业面临的核心挑战：

| 痛点 | 描述 |
|------|------|
| **多平台数据孤岛** | 巨量引擎、磁力引擎、腾讯广告、小红书聚光各自独立，人工汇总分析耗时易错 |
| **决策-执行链路断裂** | 从发现问题到跨平台执行优化，需多工具切换，响应滞后 |
| **内容-投放-电商割裂** | 内容生产、广告投放、直播间运营由不同团队负责，缺乏统一 AI 调度中枢 |
| **客户信任缺失** | 客户需看懂 AI 在干什么、为什么这么干，才能放心放权 |

### 1.2 项目价值

AdCockpit 作为 **AI 数字营销中枢**，通过 Multi-Agent 协作将上述割裂的环节打通，
以**透明、可干预、可视化**的方式呈现 Agent 的"思考"与"行动"全过程。

**核心差异化**：不是传统的 Chatbot（一问一答），而是**对话驾驶舱**——

- 客户/运营人员用自然语言下指令
- 中间面板实时展示 8 个 Agent 的协作过程
- 右侧仪表盘动态呈现数据洞察
- 关键操作支持**人工审批中断**（Human-in-the-loop）

### 2.3 8 个 Agent 职责

| Agent | 职责 | 对应 JD 能力 |
|-------|------|-------------|
| **Supervisor** | 解析用户意图，识别场景，拆解子任务，动态路由 | Agent 编排 |
| **Data** | 并行拉取多平台 API 数据，返回统一 Schema | 工具调用/平台打通 |
| **Analysis** | 数据筛选、异常检测、多模态分析、根因诊断 | 业务理解 |
| **Strategy** | 生成优化策略（调价/调预算/再分配），含风险等级 | 业务决策 |
| **Content** | 素材分析、爆款特征提取、AI 脚本生成 | 内容生产 |
| **Ecommerce** | 库存监控、优惠券管理、催单话术、ERP 联动 | 电商闭环 |
| **Execute** | 统一操作执行、人工确认中断、失败重试 | Human-in-the-loop |
| **Report** | 聚合报告生成、PPT 提纲、数据可视化 | 数据分析 |

## 三、五大核心场景

### 场景 1：跨平台广告投放智能优化 (P1)
**用户**：优化师小王
**输入**："检查最近 7 天抖音和腾讯广告 Top 5 消耗计划中 ROI<2 的，自动降价 10%"
**Agent 流程**：Supervisor 拆解 → Data 并行拉取两平台 → Analysis 筛选 ROI<2 → Strategy 生成调价策略 → **人工审批卡片** → Execute 执行 → Report 输出报告
**UI 体现**：中间面板 7 步骤全流程可视，右侧 ROI 指标/柱状图/告警，左侧确认卡片

### 场景 2：投后素材分析与 AI 内容工厂 (P2)
**用户**：内容运营小张
**输入**："找出昨日 CTR 最高和最低的 3 条视频，分析爆款共性，生成 3 条新脚本，发布到飞书"
**Agent 流程**：Data 拉取素材 → Analysis 多模态帧分析 → Content 生成脚本 → Execute 发布飞书

### 场景 3：直播间实时 AI 场控与电商闭环 (P2)
**用户**：直播运营小刘
**输入**："查库存，低于 50 件补 200 件，创建 10 元优惠券，生成催单话术"
**Agent 流程**：Data 并行查询库存+直播指标 → Analysis 缺口评估 → Ecommerce 策略 → Execute 并行执行(补货需确认+发券自动)

### 场景 4：跨渠道数据分析与预算决策 (P2)
**用户**：客户总监老李
**输入**："拉取本月抖音/腾讯/小红书数据按客户汇总，找回报率 Top3/Bottom2，生成 PPT 提纲"
**Agent 流程**：Data 并行拉取三平台 → Analysis 客户排名 → Strategy 预算建议 → Report PPT 提纲

### 场景 5：全链路故障诊断与自主恢复 (P3)
**用户**：优化师小王
**输入**："计划 12345 没量了，排查原因，出价问题提价 5%，审核拒绝替换素材并通知我"
**Agent 流程**：Data 查状态 → Analysis 根因分析 → Strategy 恢复方案 → Execute 自动恢复(替换素材+重提审+通知)

## 四、技术栈与关键设计

### 4.1 技术选型

| 层次 | 技术 | 选择理由 |
|------|------|---------|
| 前端 | **Streamlit** | 纯 Python 快速原型，三栏布局 + 自定义 CSS 注入，无需前端构建工具 |
| 后端 | **FastAPI** | 高性能异步框架，原生 SSE 支持，Pydantic 数据校验 |
| Agent 编排 | **LangGraph StateGraph** | 有状态图执行，原生 `interrupt()` HITL 支持，MemorySaver 持久化 |
| LLM 接口 | **LangChain + langchain-openai** | 统一 Function Calling 接口，支持多模型切换 |
| 数据模拟 | **Mock 函数 (24个)** | 10% 业务异常 + 5% 网络异常概率，模拟真实 API |
| 实时通信 | **SSE (Server-Sent Events)** | 单向推送 Agent 状态变更，比 WebSocket 轻量 |

### 4.2 Human-in-the-loop 机制

```
Strategy Agent 生成策略 → Execute Agent 检查 requires_approval
  ┌─ True → pending_approval 设值 → LangGraph interrupt() 暂停
  │         前端渲染确认卡片 → 用户确认/拒绝
  │         确认 → POST /approve → Command(resume=...) → 继续执行
  │         拒绝 → POST /reject  → 操作取消 + 状态回滚
  └─ False → 直接执行（低风险操作）
```

### 4.3 Mock 数据一致性设计

所有 Mock 与 HTML 原型、Dashboard 图表共享同一数据源（`tools/mock_data.py`），确保：
- 原型 HTML → 静态展示与 Demo 一致
- Streamlit Dashboard → 动态读取与 Mock 函数一致
- pytest 测试 → 断言值与原型一致

### 4.4 LangGraph State (核心数据结构)

```python
class AgentState(TypedDict):
    user_input: str                    # 用户原始输入
    task_graph: List[TaskNode]         # 任务 DAG
    platform_data: List[PlatformData]  # 多平台数据
    analysis_result: Dict              # 分析结果
    strategy_actions: List[StrategyAction]  # 策略列表
    execution_results: List[ExecutionResult] # 执行结果
    report: Dict                       # 报告
    pending_approval: Optional[List]   # HITL 审批队列
    current_scene: Literal[...]        # 场景路由
```

## 五、开发方式：AI Coding 工作流

本项目 95% 以上代码由 **Claude Code** 生成，完整践行了 JD 中要求的 AI Coding 范式：

```
Spec (需求规格) → Plan (实施计划) → Tasks (任务拆分) → Implement (AI 生成代码) → Verify (验证)
```

| 阶段 | 工具/命令 | 产出 |
|------|----------|------|
| 需求定义 | `/speckit-specify` | spec.md — 5 个用户场景 + 44 条功能需求 |
| 架构规划 | `/speckit-plan` | plan.md — 5 阶段 37 文件规划 + 14 条 AI 提示词 |
| 任务拆分 | `/speckit-tasks` | tasks.md — 45 个可执行任务 + 推荐提示词 |
| 代码实现 | `/speckit-implement` | 32 个 Python 文件 + 1 个 HTML 原型 |
| 快速迭代 | 自然语言对话 | 2 天内完成从概念到可运行 Demo |
