## 一、理解核心概念

### 你和飞书之间，有一个"中间人"

```
你（用户）                   飞书平台
   │                           │
   │   你打开飞书App            │
   │   手动创建文档              │
   └───────────────────────────┘
   这是"人操作飞书"的模式


AdCockpit（程序）     飞书应用（中间人）       飞书平台
   │                      │                    │
   │  通过API创建文档      │  代表谁？有权限吗？ │
   └──────────────────────┼────────────────────┘
   这是"程序操作飞书"的模式 —— 需要一个"中间人"
```

**为什么需要"中间人"？**

飞书不允许任意程序直接操作你的文档。你需要：
1. 在飞书开放平台注册一个"应用"（就是中间人的身份）
2. 给这个应用授权（允许它操作哪些东西）
3. 程序带上应用的凭证去调用飞书 API

### 三个关键角色

| 角色 | 是什么 | 在哪 |
|------|--------|------|
| **飞书应用** | 你注册的"机器人"，代表你的程序去操作飞书 | open.feishu.cn 创建 |
| **APP_ID** | 应用的身份证号，告诉飞书"我是谁" | 应用页面 → 凭证与基础信息 |
| **APP_SECRET** | 应用的密码，证明"我真的是我" | 同上（不能泄露） |

### 类比：银行 ATM

```
APP_ID     = 你的银行卡号（公开的，识别你是谁）
APP_SECRET = 你的银行卡密码（保密的，证明是你本人）
飞书应用   = 你在银行开的账户
API 调用   = 通过 ATM 取钱
权限       = 你的账户开通了哪些功能（转账？理财？）
```

## 三、代码怎么用凭证

### 调用链路

```
Streamlit 点击按钮
       │
       ▼
chat_panel.py: 调用 publish_scripts()
       │
       ▼
feishu_client.py: publish_scripts()
       │
       ├─ 读取 .env.example 拿到 APP_ID + APP_SECRET
       │
       ├─ 用 APP_ID + APP_SECRET 换取 tenant_access_token（临时通行证）
       │     POST /auth/v3/tenant_access_token/internal
       │     {app_id, app_secret} → {tenant_access_token: "t-xxxx"}
       │
       ├─ 用 token 创建文档
       │     POST /docx/v1/documents
       │     Authorization: Bearer t-xxxx
       │     {title: "脚本1"}
       │     → {document_id: "xxx", url: "https://..."}
       │
       └─ 返回文档链接给用户
```

### 代码位置

| 文件 | 做什么 |
|------|--------|
| `.env.example` | 存放 APP_ID 和 APP_SECRET（凭证） |
| `tools/feishu_client.py:689` | `load_dotenv()` 读取凭证 |
| `tools/feishu_client.py:_get_tenant_token()` | 用凭证换临时 token |
| `tools/feishu_client.py:create_document()` | 创建文档 |
| `tools/feishu_client.py:publish_scripts()` | 批量创建脚本文档 |

## 四、权限是怎么回事

### 你的应用目前有哪些权限

飞书应用创建后**默认什么权限都没有**。需要手动开通：

| 我们需要的权限 | 你应用是否已开通 | 没有的话会怎样 |
|-------------|:---:|------|
| `docx:document`（创建文档） | ✅ 是 | 无法创建文档 |
| `docx:document:create`（写入内容） | ✅ 是 | 文档是空的 |
| `drive:drive`（访问云文档空间） | ❌ 否 | **文档不会出现在文件夹里** |

### 为什么文档创建了但文件夹里看不到

这就是 `drive:drive` 权限缺失导致的：

```
没有 drive:drive 权限：
  文档创建成功 → 存放在应用的独立空间 → 通过链接可以访问 ✅
                                          → 你的个人文件夹看不到 ❌

有 drive:drive 权限：
  文档创建成功 → 放到你指定的文件夹 → 文件夹里就能看到 ✅
```

### 如何开通 drive:drive 权限

> ⚠️ 这步需要飞书管理员权限（如果你在测试企业里，你自己就是管理员）

1. 打开 https://open.feishu.cn → 找到你的应用
2. 左侧「权限管理」→ 搜索 `drive:drive`
3. 开通 → 发布新版本

## 六、FAQ

**Q: 我完全忘了是谁、什么时候创建的这个应用？**
A: APP_ID 以 `cli_` 开头，说明是"企业自建应用"。当初应该是你（或同事）在 open.feishu.cn 创建的。只要能登录 open.feishu.cn，就能看到这个应用。

**Q: 怎么看我的应用？**
A: 访问 https://open.feishu.cn → 登录 → 左侧「开发者后台」→ 就能看到你创建过的所有应用。

**Q: 凭证安全吗？要不要加密？**
A: APP_SECRET 相当于密码，不要上传到公开的 GitHub 仓库。当前的 `.env.example` 已经在 `.gitignore` 里了（不会被提交）。

**Q: 我能不能用 Postman 测试？**
A: 可以：
```bash
# 1. 获取 token
POST https://open.feishu.cn/open-apis/auth/v3/tenant_access_token/internal
Body: {"app_id":"你的ID","app_secret":"你的Secret"}

# 2. 创建文档
POST https://open.feishu.cn/open-apis/docx/v1/documents
Headers: Authorization: Bearer <上一步拿到的token>
Body: {"title":"测试文档"}
```
