## 一、Multi-Agent 编排框架

### Q1：你是如何设计 Multi-Agent 系统的？请介绍架构和设计思路。

**参考回答**：

我设计的 AdCockpit 系统包含 8 个专业 Agent，基于 LangGraph StateGraph 进行编排。

核心设计思路：
1. **单一职责** — 每个 Agent 只负责一个专业领域，如 Data Agent 只管数据拉取，Strategy Agent 只管策略生成
2. **Supervisor 路由** — 通过 Supervisor Agent 识别用户意图，动态路由到不同的下游 Agent 链
3. **并行执行** — Data Agent 可同时并行调用抖音和腾讯两个平台的 API，减少总耗时
4. **共享状态** — 所有 Agent 共享一个 `AgentState` TypedDict，通过 LangGraph 的图节点传递

拓扑结构：
```
Supervisor → Data Agent (并行) → Analysis Agent
→ route_after_analysis (条件分支)
→ Strategy Agent | Content Agent | Ecommerce Agent
→ Execute Agent (HITL 中断点)
→ Report Agent
```

**与 JD 对应**：Multi-Agent 编排框架、工具调用链路、记忆与状态管理。

---

### Q2：LangGraph 相比其他 Agent 框架有什么优势？

**参考回答**：

| 特性 | LangGraph | CrewAI | LangChain Agent |
|------|-----------|--------|-----------------|
| 图执行模型 | StateGraph + 条件边 | 顺序/层级 | ReAct 循环 |
| HITL 支持 | 原生 `interrupt()` | 需手动实现 | 不支持 |
| 状态持久化 | MemorySaver 内置 | 有限 | 需额外实现 |
| 并行执行 | 原生支持 | 有限 | 不支持 |
| 生产就绪度 | 高（LangSmith 集成） | 中 | 中 |

在 AdCockpit 中，我选择 LangGraph 的关键原因是：
1. **Human-in-the-loop**：`interrupt()` 可以精确地在 Execute Agent 前暂停，等待用户确认
2. **条件路由**：`add_conditional_edges()` 根据 `current_scene` 自动路由到不同 Agent
3. **状态管理**：所有数据通过 `AgentState` 在节点间流转，MemorySaver 支持中断恢复

## 二、工具调用与 Function Calling

### Q3：你是如何设计和实现 Agent 的工具调用链路的？

**参考回答**：

工具调用链路分为三层：

**1. 工具定义层**（`tools/mock_functions.py`）
- 24 个 Mock 函数按业务域分组：广告投放 8 个 + 内容生产 4 个 + 电商闭环 5 个 + 数据分析 3 个 + 通知系统 4 个
- 每个函数包含完整的 docstring、参数校验和异常模拟
- 使用 10% 业务异常 + 5% 网络异常的概率模拟真实环境

**2. 工具注册层**（`tools/tool_registry.py`）
- 使用 LangChain `@tool` 装饰器将 Mock 函数封装为标准 Tool
- 按域名分组导出 `AD_TOOLS`、`CONTENT_TOOLS`、`ECOM_TOOLS`、`DATA_TOOLS`、`NOTIFY_TOOLS`
- Agent 节点根据 `current_scene` 选择对应的工具集

**3. 执行与展示层**
- Agent 调用工具时，中间面板实时展示 Request/Response Schema
- 工具调用详情可展开查看，包括耗时和状态
- 异常自动重试（最多 3 次），失败记录到 `error_log`

**与 JD 对应**：打通各平台 API、构建 Agent 的工具生态。

### Q4：请解释 Function Calling 在 Agent 系统中的工作原理。

**参考回答**：

Function Calling 是 LLM 调用外部工具的核心机制。工作流程：

1. **定义工具 Schema**：用 Pydantic 模型定义函数签名（参数类型、描述、约束）
2. **注入 Prompt**：将工具定义作为 System Prompt 的一部分传给 LLM
3. **LLM 决策**：LLM 判断是否需要调用工具，如需则输出结构化的函数调用 JSON
4. **执行工具**：Agent 框架解析 LLM 的函数调用，执行实际函数
5. **结果注入**：将工具返回结果注入对话上下文，LLM 继续推理

在 AdCockpit 中：
- 使用 LangChain `@tool` 装饰器定义工具
- 通过 `StructuredTool.from_function()` 封装
- Pydantic `BaseModel` 定义 `args_schema` 确保类型安全
- Mock 函数模拟真实 API 响应，包含异常场景

## 三、Human-in-the-loop

### Q5：你的系统中如何实现 Human-in-the-loop？请描述具体的审批中断流程。

**参考回答**：

AdCockpit 使用 LangGraph 原生的 `interrupt()` 机制实现 Human-in-the-loop。

**具体流程**：

1. **策略生成**：Strategy Agent 生成 `StrategyAction` 列表，每条标注 `risk_level` 和 `requires_approval`
2. **中断判断**：Execute Agent 检查是否有 `requires_approval=True` 的动作
3. **暂停执行**：将待审批 action 放入 `pending_approval`，调用 `interrupt()` 暂停图
4. **用户确认**：前端渲染确认卡片，展示操作摘要/影响/风险等级，用户点击确认或取消
5. **恢复执行**：通过 `POST /approve` 端点发送 `Command(resume={'approved': True})` 恢复图执行
6. **超时处理**：默认 5 分钟无响应自动视为拒绝

**代码示例**（简化版）：
```python
# Execute Agent 中
if any(a["requires_approval"] for a in actions):
    state["pending_approval"] = pending_actions
    interrupt(pending_actions)  # 暂停，等待外部信号

# 恢复端点
@app.post("/approve/{session_id}")
def approve(session_id):
    graph.invoke(Command(resume={"approved": True}), config)
```

**与 JD 对应**：记忆与状态管理、Agent 编排框架核心能力。

## 四、AI Coding 工作流

### Q6：你如何使用 AI Coding 完成这个项目的？请描述你的工作流程。

**参考回答**：

我遵循 **Spec → Plan → Tasks → Implement** 的 AI Coding 工作流：

**1. Spec 阶段**（需求规格）
- 使用 `/speckit-specify` 命令，基于业务需求生成完整的 feature spec
- 输出包含 5 个 User Story（带优先级和验收场景）、44 条功能需求、7 条成功标准

**2. Plan 阶段**（实施计划）
- 使用 `/speckit-plan` 生成实施计划，包含技术选型、模块拆分、依赖关系
- 每个阶段附带**可直接在 Claude Code 中使用的实现提示词**（14 条）

**3. Tasks 阶段**（任务拆分）
- 使用 `/speckit-tasks` 拆分为 45 个可执行任务
- 每个任务标注文件路径、并行标记 [P]、用户故事标签 [US1-US5]

**4. Implement 阶段**（代码实现）
- 使用 `/speckit-implement` 自动执行任务列表
- 32 个 Python 文件全部由 AI 生成，覆盖 agents/api/tools/ui 全部模块

**关键体会**：
- AI Coding 的核心在于**提示词质量**和**上下文管理**
- Spec/Plan/Tasks 越详细，AI 生成的代码质量越高
- 人工 Review 聚焦在业务逻辑合理性和架构一致性上

**与 JD 对应**：熟练使用 AI Coding 工具，能用 AI 完成 95% 以上编码工作。

---

### Q7：AI Coding 中遇到过什么挑战？如何解决的？

**参考回答**：

几个典型挑战：

1. **上下文限制**：复杂系统无法一次放进 context。**解决**：用 Spec→Plan→Tasks 分层拆分，每层抽象度不同

2. **一致性保证**：多次 AI 调用生成的代码风格不一致。**解决**：宪法文件（constitution.md）约束全局规则，Mock 数据抽取到独立文件作为单一事实源

3. **CSS/UI 精确度**：AI 生成的 CSS 与设计稿有偏差。**解决**：先用 AI 生成完整 HTML 原型（`adcockpit-prototype.html`）作为参考标准，再让 AI 对照原型实现 Streamlit 版本

4. **版本依赖**：LangGraph 1.0+ 的 API 与旧版不兼容。**解决**：在宪法中锁定版本号，代码中增加 `try/except ImportError` 降级逻辑

## 五、业务理解

### Q8：你如何理解数字营销的业务逻辑？在 AdCockpit 中是如何体现的？

**参考回答**：

数字营销的核心链路：

```
广告投放 → 素材/内容 → 直播/电商转化 → 数据分析 → 优化迭代
```

在 AdCockpit 中，每个环节都有对应的 Agent 和工具：

| 业务环节 | Agent | Mock 函数示例 |
|---------|-------|-------------|
| 广告投放 | Data + Analysis + Strategy | `get_top_campaigns`, `update_bid`, `update_budget` |
| 素材内容 | Content + Analysis | `get_top_creatives`, `analyze_video_frames`, `generate_script` |
| 电商转化 | Ecommerce + Execute | `get_product_stock`, `update_stock`, `create_coupon` |
| 数据分析 | Report + Strategy | `get_multi_platform_report`, `generate_budget_proposal` |

**关键业务指标**：
- **ROI** = 收入/消耗，健康线通常 > 2.0
- **CPA** = 消耗/转化数，衡量获客成本
- **CTR** = 点击数/展示数，衡量素材吸引力
- **CVR** = 转化数/点击数，衡量落地页效果

这些指标在 Dashboard 中实时展示，低于阈值的计划自动告警。

## 六、记忆与状态管理

### Q9：你的 Agent 系统如何管理记忆和状态？

**参考回答**：

AdCockpit 采用分层记忆架构：

**短期记忆**（对话上下文）：
- `conversation_history` 保留最近 20 轮对话
- 每次对话轮次追加到 `AgentState` 中
- 用于多轮对话澄清和上下文理解

**长期记忆**（用户偏好）：
- `save_user_preference / load_user_preference` Mock 函数
- 存储用户偏好：默认 ROI 阈值、关注平台、模板偏好等
- 跨会话持久化（内存中，可扩展为 JSON 文件或数据库）

**状态管理**（LangGraph）：
- `AgentState` TypedDict 作为全局状态
- LangGraph `MemorySaver` 支持 checkout/checkpoint
- 中断后可从断点恢复执行

**与 JD 对应**：记忆与状态管理模块。

## 七、平台 API 打通

### Q10：你如何设计 Mock Server 来模拟真实平台 API？

**参考回答**：

Mock Server 设计遵循"易于替换为真实 API"的原则：

**1. 独立部署**：`mock_server/main.py` 是独立的 FastAPI 应用，端口 8100，
与主 Agent 后端（端口 8000）解耦

**2. RESTful 规范**：所有端点遵循 `GET/POST /api/{platform}/{resource}` 格式，
与真实平台 API 的路径结构一致

**3. 真实 Schema**：Request/Response 使用 Pydantic 模型定义，字段与真实 API 对齐
```python
@router.get("/api/douyin/campaigns/top")
async def douyin_get_top_campaigns(days: int=7, top_n: int=5):
    # 真实环境只需替换函数体，接口签名不变
    return {"code": 0, "data": mock_data}
```

**4. 异常模拟**：5% 概率返回 503（网络异常），10% 概率返回 500（业务异常），
测试 Agent 的鲁棒性

**5. 数据一致性**：Mock Server 和 Mock Functions 共享 `tools/mock_data.py` 数据源

**与 JD 对应**：打通各平台 API（巨量引擎、腾讯广告、小红书聚光等）。
