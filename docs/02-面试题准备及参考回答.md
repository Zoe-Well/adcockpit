# AdCockpit v1.0 面试题准备及参考回答

> 按 JD 关键要求组织，每道题结合 AdCockpit 实际实现。

## 一、Multi-Agent 编排框架

### Q1：你是如何设计 Multi-Agent 系统的？

AdCockpit 包含 8 个专业 Agent（Supervisor/Data/Analysis/Strategy/Content/Ecommerce/Execute/Report），基于 LangGraph StateGraph 编排。

核心设计决策：
1. **单一职责** — 每个 Agent 只做一件事
2. **Supervisor 路由** — 中央调度器识别意图，分派给对应 Agent
3. **并行执行** — Data Agent 内部用 `ThreadPoolExecutor` 并行调用多平台 API
4. **共享 State** — `AgentState` TypedDict 在节点间流转，MemorySaver 支持中断恢复
5. **条件分支** — `add_conditional_edges()` 根据 `current_scene` 路由到 Strategy/Content/Ecommerce

### Q2：LangGraph 相比其他框架的优势？

| 特性 | LangGraph | CrewAI |
|------|-----------|--------|
| 图执行 | StateGraph + 条件边 | 顺序/层级 |
| HITL | 原生 `interrupt()` | 需手动实现 |
| 状态持久化 | MemorySaver 内置 | 有限 |
| 并行执行 | 原生支持 | 有限 |

AdCockpit 选择 LangGraph 的核心原因：`interrupt()` 可以在 Execute Agent 前精确暂停，等待用户确认后再恢复。

## 二、Human-in-the-loop

### Q3：如何实现审批中断？

AdCockpit 的三步确认机制：

1. **参数面板**：执行前用户可以调整所有参数（平台/阈值/比例）
2. **流式动画 + 确认卡片**：动画展示 Agent 思考过程，执行步骤停住等待确认
3. **确认后才执行**：点击"确认执行"才真正调用 `update_bid`/`update_budget`，取消则无任何影响

投放优化和投放创建都需要确认；内容生产需要预览后确认才发布飞书。

### Q4：为什么不是纯自动化？

作为 B 端服务商，客户预算是你操作的。全自动 = 不可控 = 不信任。让客户/运营看到每一步、确认每一项，比快更重要。

## 三、AI Coding 工作流

### Q5：你如何使用 AI Coding 完成本项目？

**Spec → Plan → Tasks → Implement** 四阶段：

1. **Spec**：`/speckit-specify` 生成 5 个 User Story + 44 条功能需求
2. **Plan**：`/speckit-plan` 生成技术选型 + 37 文件规划 + 14 条 AI 提示词
3. **Tasks**：`/speckit-tasks` 拆分为 45 个可执行任务
4. **Implement**：`/speckit-implement` 逐任务生成代码

**关键体会**：
- 提示词质量决定代码质量 — Spec/Plan/Tasks 越详细越好
- 宪法文件（constitution.md）约束全局规则，保证一致性
- HTML 原型作为 UI 标准，AI 对照原型实现

### Q6：95% AI Coding 的数据从哪来？

项目 32+ Python 文件，其中 30 个完全由 AI 生成（agents/*, tools/*, ui/*, api/*, mock_server/*）。人工编写的只有少量配置文件。所有业务逻辑、UI 组件、动画引擎均通过自然语言描述让 AI 实现。

## 四、工具调用与 Function Calling

### Q7：如何实现 Agent 的工具调用？

三层架构：

1. **Mock 函数层**（`tools/mock_functions.py`）：24 个函数模拟 6 个平台 API，10%/5% 异常概率
2. **工具注册层**（`tools/tool_registry.py`）：LangChain `@tool` 装饰器封装，按域分组
3. **执行层**（`tools/feishu_client.py`）：真实飞书 API 集成，获取 token → 创建文档 → 写入内容

每个工具调用在 Trace Board 中展示参数和结果，异常自动重试（最多 5 次）。

## 五、真实集成 vs Mock

### Q8：飞书集成是怎么做的？

**真实 API**：使用飞书 Open API（`docx/v1/documents` + `children` 端点），OAuth 获取 tenant_access_token，创建文档并写入内容。

**Mock 降级**：未配置凭证时自动降级为 Demo 模式，返回模拟链接，不影响其他功能。

**为什么不接 OAuth 用户授权**：Demo 阶段应用级 Token 足够，生产环境需 OAuth 让文档归属用户。

## 六、业务理解

### Q9：你如何理解数字营销指标？

| 指标 | 含义 | 优化方向 |
|------|------|---------|
| ROI | 收入/消耗 | <2.0 需调价或降预算 |
| CPA | 消耗/转化数 | 高则优化落地页 |
| CTR | 点击数/展示数 | 低则换素材 |
| CVR | 转化数/点击数 | 低则发券刺激 |

AdCockpit 的 Dashboard 实时展示这些指标，低于阈值的计划自动高亮告警。

## 七、记忆与状态管理

短期记忆（`conversation_history`）保留最近 20 轮对话；长期记忆（`save/load_user_preference`）存储用户偏好；工作记忆（AgentState）在 LangGraph 节点间流转。

状态持久化：`MemorySaver` checkpoint 支持中断后恢复，避免审批超时丢失进度。
