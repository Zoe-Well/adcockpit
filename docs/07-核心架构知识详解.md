## 一、Web 基础

### 1.1 前端 vs 后端：一个通俗的类比

把 Web 应用想象成一家餐厅：

```
客户（浏览器）        服务员（前端）         厨师（后端）
   │                      │                    │
   │ "我要一份牛排"       │                    │
   │─────────────────────→│                    │
   │                      │  "8号桌要牛排"     │
   │                      │───────────────────→│
   │                      │                    │ 烹饪中...
   │                      │    "牛排好了"      │
   │                      │←───────────────────│
   │  "您的牛排来了"      │                    │
   │←─────────────────────│                    │
```

- **前端（Frontend）**：用户看到和交互的界面——按钮、输入框、动画、表格。跑在浏览器里。
- **后端（Backend）**：处理业务逻辑、读写数据、调用 AI。跑在服务器上。
- **API（Application Programming Interface）**：前后端之间的"传菜窗口"——前端通过 HTTP 请求告诉后端要做什么，后端返回结果。

### 1.2 HTTP 请求-响应模型

```
客户端发请求 → 服务器处理 → 返回响应
```

```
请求（Request）：
  POST /api/intent/classify          ← URL 路径（要做什么）
  Content-Type: application/json     ← 头信息（什么格式）
  {"user_input": "优化投放ROI"}       ← 请求体（具体内容）

响应（Response）：
  200 OK                             ← 状态码（成功）
  {"type": "business", "scene": "ad_placement", "reply": "..."}  ← 响应体
```

**常见 HTTP 方法**：

| 方法 | 含义 | AdCockpit 示例 |
|------|------|---------------|
| `GET` | 获取数据 | `GET /api/campaigns/all` → 加载所有计划 |
| `POST` | 提交数据/触发操作 | `POST /api/agent/optimize` → 执行优化 |
| `PUT` | 更新数据 | 更新计划信息 |
| `DELETE` | 删除数据 | 删除计划 |

**常见状态码**：

| 状态码 | 含义 |
|--------|------|
| `200 OK` | 成功 |
| `201 Created` | 创建成功 |
| `400 Bad Request` | 请求参数有误 |
| `401 Unauthorized` | 未认证（API Key 无效） |
| `404 Not Found` | 资源不存在 |
| `500 Internal Server Error` | 服务器内部错误 |

### 1.3 JSON（JavaScript Object Notation）

前后端通信的数据格式，人类可读、机器易解析：

```json
{
  "type": "business",
  "scene": "ad_placement",
  "reply": "识别到你想优化广告投放效果。",
  "confidence": 0.94
}
```

支持 6 种类型：字符串 `"..."`、数字 `1.5`、布尔 `true/false`、数组 `[...]`、对象 `{...}`、null。

## 二、前端技术栈

AdCockpit 前端是一个**单页应用（SPA — Single Page Application）**。
不刷新页面，所有交互在一个 HTML 里完成，通过 JS 动态切换内容。

### 2.1 React 18 — JavaScript UI 框架

**核心概念**：React 把 UI 拆成**组件（Component）**——可复用的 UI 积木块。

```
页面 = 组件树
App
├── Header（顶部导航栏）
├── Sidebar（左侧菜单）
│   ├── NavButton × 6（标签按钮）
│   └── UserProfile（用户头像）
├── ChatPanel（对话面板）
│   ├── Message × N（消息列表）
│   ├── ParamCard（参数设置卡片）
│   └── InputBar（输入框）
├── TracePanel（Trace Board）
│   └── StepCard × N（步骤卡片）
└── Dashboard（数据仪表盘）
    ├── MetricCard × 3（指标卡）
    ├── BarChart（柱状图）
    └── DataTable（数据表格）
```

**核心机制 —— 状态（State）与渲染**：

```tsx
// 当调用 setTab('content') 时，React 自动重新渲染相关组件
const [tab, setTab] = useState('optimize')  // 当前选中的标签

// JSX：在 JavaScript 里写 HTML
<button onClick={() => setTab('content')}>
  内容生产
</button>
```

**核心机制 —— 副作用（Effect）**：

```tsx
// 当组件首次加载时，自动执行 loadPlans()
useEffect(() => { loadPlans() }, [])
```

### 2.2 TypeScript — JavaScript + 类型

JavaScript 是动态类型语言，容易在运行时出错。TypeScript 加了类型系统，**在写代码时就发现类型错误**：

```typescript
// JavaScript（无类型）
function runOptimize(params) { ... }  // params 是什么？不知道

// TypeScript（有类型）
type Tab = 'optimize' | 'create' | 'content' | 'ecommerce'
function runOptimize(params: OptimizeParams): Promise<void> { ... }
// 参数类型明确，返回值类型明确，编辑器能自动补全和报错
```

### 2.3 Tailwind CSS — 原子化 CSS

传统 CSS：写一个类名 → 在 CSS 文件里定义样式。
Tailwind：直接在 HTML 里用工具类名组合样式，**所见即所得**：

```html
<!-- 传统写法 -->
<div class="card"></div>
<style>.card { background: white; border-radius: 8px; padding: 12px; }</style>

<!-- Tailwind 写法 -->
<div class="bg-white rounded-lg p-3">
  <!-- bg-white = 白色背景, rounded-lg = 大圆角, p-3 = padding -->
</div>
```

**优点**：不需要在 HTML 和 CSS 文件间跳转，改样式就是改类名。类似乐高积木——每个类一个功能，组合使用。

### 2.4 shadcn/ui — React 组件库

提供 Button、Card、Input、Badge 等常用组件的**源码**（不是 npm 包），可以直接修改。类似给了你一套设计好的组件模板，可以完全自定义。

```tsx
import { Button } from './components/ui/button'
import { Card, CardContent } from './components/ui/card'

<Card>
  <CardContent className="p-4">
    <Button size="sm" onClick={handleClick}>开始优化</Button>
  </CardContent>
</Card>
```

## 三、后端技术栈

### 3.1 FastAPI — Python Web 框架

FastAPI 是 Python 最快的 Web 框架之一。它的核心工作：**定义 API 端点 → 接收请求 → 处理逻辑 → 返回响应**。

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class AgentRequest(BaseModel):                             # 定义请求格式
    user_input: str
    platforms: list[str] = ["douyin", "tencent"]

@app.post("/api/agent/optimize")                           # 定义端点
async def agent_optimize(body: AgentRequest):              # 自动校验参数
    # ... 业务逻辑 ...
    return {"steps": steps, "changes": changes}
```

**为什么用 FastAPI？**
- **快**：性能接近 Node.js/Go
- **自动文档**：访问 `http://localhost:8000/docs` 即可看到所有 API 的交互式文档
- **类型安全**：Pydantic 自动校验请求参数格式

### 3.2 Pydantic — 数据类型校验

类似前端的 TypeScript，但用于 Python 后端：

```python
class AgentRequest(BaseModel):
    user_input: str                           # 必须是字符串
    platforms: list[str] = ["douyin"]          # 必须是字符串列表，默认值 douyin
    roi_threshold: float = 2.0                 # 必须是浮点数，默认 2.0

# 自动校验：如果前端传了 {"user_input": 123}，FastAPI 直接返回 422 错误
# 不用手动写 if type(x) == str: ...
```

### 3.3 CORS（跨域资源共享）

浏览器的安全策略：`localhost:5173`（前端）不能随便请求 `localhost:8000`（后端）。
需要在后端设置 CORS 允许：

```python
app.add_middleware(CORSMiddleware,
    allow_origins=["http://localhost:5173"],  # 允许哪些前端地址访问
    allow_methods=["*"],                       # 允许所有 HTTP 方法
)
```

## 四、AI Agent 层

这是 AdCockpit 最核心的部分，也是和传统 Web 应用最大的区别。

### 4.1 LLM（大语言模型）：DeepSeek

**LLM 是什么？**一个"超级文本预测器"——给定一段文字（prompt），预测最可能的下文。

```
输入（Prompt）：     "分析以下广告计划的ROI表现：C001 ROI=1.5, C002 ROI=2.3"
输出（Completion）：  "C001 的 ROI 低于健康线 2.0，建议降价 10%；C002 表现正常。"
```

**DeepSeek**：国产 LLM，性价比高，API 兼容 OpenAI 格式。AdCockpit 通过 OpenAI SDK 调用：

```python
from openai import OpenAI
client = OpenAI(api_key="sk-xxx", base_url="https://api.deepseek.com/v1")

response = client.chat.completions.create(
    model="deepseek-chat",
    messages=[{"role": "system", "content": "你是广告数据分析师..."},
              {"role": "user", "content": "分析这些计划..."}],
    temperature=0.3,   # 随机性 0-2，越低越确定
    max_tokens=500,     # 最大输出长度
)
```

**关键参数**：
- `temperature`：0.1 = 几乎每次一样（适合分类/数据提取），0.9 = 每次不同（适合创意生成）
- `max_tokens`：限制输出长度，防止浪费

### 4.2 Agent 模式 vs 传统 Prompt

**传统 Prompt**：一次问答。

```
用户："优化我的广告"
LLM："建议你降价 10%，降预算 20%"
```

问题：LLM 不能真正查询数据、不能执行操作——只能"说"。

**Agent 模式**：LLM 作为"大脑"调度"工具"。

```
用户："优化我的广告"
    │
    ▼
Supervisor Agent（大脑）："我需要先查数据，再分析，再执行"
    │
    ├──→ Data Agent（工具）：调用 get_top_campaigns() 拉取真实数据
    ├──→ Analysis Agent（大脑）：分析数据，找出异常
    ├──→ Strategy Agent（大脑）：生成优化策略
    ├──→ Execute Agent（工具）：调用 update_bid() 真正修改出价
    └──→ Report Agent（大脑）：汇总报告
```

**关键区别**：Agent 能**实际做事**（调用函数），不只是说。

### 4.3 LangGraph — Agent 编排框架

**问题**：多个 Agent 怎么协作？谁先谁后？出错怎么办？

**LangGraph** 用**有向图（Graph）**来编排 Agent 的执行顺序：

```
        ┌─────────┐
        │  START  │
        └────┬────┘
             ▼
      ┌─────────────┐
      │ supervisor   │  ← 1. LLM 理解需求、拆解任务
      └──────┬──────┘
             ▼
      ┌─────────────┐
      │ data_agent   │  ← 2. 拉取投放数据
      └──────┬──────┘
             ▼
      ┌─────────────┐
      │analysis_agent│  ← 3. LLM 分析异常
      └──────┬──────┘
             ▼
      ┌─────────────┐
      │strategy_agent│  ← 4. LLM 生成策略
      └──────┬──────┘
             ▼
      ┌─────────────┐
      │execute_agent │  ← 5. 执行调价（HITL 审批）
      └──────┬──────┘
             ▼
      ┌─────────────┐
      │report_agent  │  ← 6. 汇总报告
      └──────┬──────┘
             ▼
        ┌─────────┐
        │   END   │
        └─────────┘
```

**为什么用 LangGraph 而不是硬编码？**
- 支持**条件分支**（不同场景走不同路径）
- 支持**中断-恢复**（HITL 审批：停在 execute 等用户确认）
- 支持**状态管理**（每个节点可以读写共享状态）

### 4.4 HITL（Human-in-the-Loop，人机协同）

不是所有操作都应该自动执行。**降价 30% 是高风险的**，应该让人确认。

AdCockpit 的 HITL 模式：
1. Agent 执行到 strategy → 生成优化方案 → **停在 execute**
2. 前端显示审批卡片："T001 降价 10%、C001 降价 10%……确认执行？"
3. 用户点「确认」→ 继续执行
4. 用户点「取消」→ 不执行，流程结束

这保证了 **AI 提建议，人做决策**。

### 4.5 意图分类（Intent Classification）

在调用 LangGraph 之前，需要先知道用户想干什么。这就是**意图分类**：

```
用户输入："帮我写3条带货脚本"
    │
    ▼
意图分类器（LLM + 关键词回退）：
    → type: "business"
    → scene: "content"
    │
    ▼
前端根据 scene 跳转到对应标签页
后端根据 scene 调用对应 Agent 流程
```

AdCockpit 采用**两级分类机制**：
1. **DeepSeek LLM**（优先）—— 3 次重试
2. **Keyword Fallback**（降级）—— LLM 不可用时的关键词兜底

## 五、数据持久化

### 5.1 为什么需要持久化？

没有持久化：服务器重启 → 所有数据丢失。就像玩游戏没存档。

### 5.2 JSON 文件存储（当前方案）

```python
# backend/data/plans_douyin.json
[
  {"id": "C001", "name": "夏季促销-A", "roi": 1.5, "cost": 15200, "bid": 25},
  {"id": "C002", "name": "新品首发-B", "roi": 2.3, "cost": 14100, "bid": 30}
]
```

**优点**：零配置、人类可读、适合 Demo/小规模数据。
**缺点**：高并发下可能冲突、不适合大规模查询。

### 5.3 Redis（计划中）

**Redis** 是**内存数据库**——数据存在 RAM 里，读写极快（微秒级）。

在 AdCockpit 中的用途：
- **消息队列（Pub/Sub）**：Agent 执行进度通过 Redis 频道推送给前端
- **缓存**：缓存热点数据，减少重复计算
- **Celery 任务队列**：异步执行耗时任务（如批量调价）

### 5.4 Chroma + RAG（计划中）

**RAG（Retrieval-Augmented Generation，检索增强生成）**：
让 LLM 能"记住"私有知识。

```
传统 LLM：只能基于训练数据回答，不知道你的具体业务
RAG LLM：先把相关文档检索出来 → 拼入 prompt → LLM 基于文档回答
```

**Chroma** 是向量数据库——把文本转成数学向量，支持"语义搜索"：

```
"ROI 低于 2 的计划怎么优化？"
    │
    ▼ 向量化
[0.23, -0.45, 0.78, ...]
    │
    ▼ 在向量空间找最相似的文档
历史优化记录 #42："ROI<2 时先降出价 10%，观察 24h 后再决定是否降预算"
    │
    ▼ 拼入 prompt
LLM："根据历史经验，建议先降价 10%..."
```

## 六、实时通信

### 6.1 传统 HTTP 的局限

```
客户端：有新消息吗？
服务器：没有。
客户端：有新消息吗？（1秒后）
服务器：没有。
客户端：有新消息吗？（2秒后）
服务器：有了！Agent 步骤已完成。
```

这叫**轮询（Polling）**——客户端反复问，大部分时候白问。浪费资源、延迟高。

### 6.2 WebSocket（全双工）

```
客户端 ═══════════════════ 服务器
        建立连接后，双方随时可以发消息

服务器 → 客户端："step_start: supervisor 开始执行"
服务器 → 客户端："step_complete: supervisor 完成"
服务器 → 客户端："step_start: data_agent 开始执行"
...
```

一个连接，双向通信，服务器主动推送。适合**实时 Trace Board**。

AdCockpit 的 WebSocket 端点：
```python
@router.websocket("/ws/{session_id}")
async def websocket_endpoint(websocket, session_id):
    # 订阅 Redis 频道
    # 有新消息 → 推送给前端
```

### 6.3 SSE（Server-Sent Events，服务器推送事件）

WebSocket 的"轻量版"——服务器主动推送，但只支持**单向**（服务器→客户端）。
适合只需推送进度的场景。在 HTTP 协议上运行，兼容性更好。

```
GET /stream/{session_id}
  → event: step_start
    data: {"node": "supervisor", "ts": "2026-06-17T10:30:00"}
  → event: step_complete
    data: {"node": "supervisor", "status": "done"}
```

## 七、AdCockpit 完整架构串讲

### 7.1 一次完整的投放优化流程

```
1. 用户输入 "优化投放ROI"
   │
   ▼
2. React → POST /api/intent/classify
   {"user_input": "优化投放ROI"}
   │
   ▼
3. FastAPI → DeepSeek LLM 分类
   返回 {"type":"business", "scene":"ad_placement"}
   │
   ▼
4. React → setTab('optimize') → 显示优化参数面板
   │
   ▼
5. 用户调整参数 → 点击「开始优化」
   │
   ▼
6. React → POST /api/agent/optimize
   │
   ▼
7. FastAPI → LangGraph 编排：
   ├── supervisor_node(LLM)     → "识别为投放优化，拆解 5 个任务"
   ├── data_agent_node(Mock)    → 拉取抖音 5 条 + 腾讯 5 条计划
   ├── analysis_agent_node(LLM) → "发现 4 条异常，2 条严重"
   ├── strategy_agent_node(LLM) → "T001 降 10%，C001 降 10%..."
   ├── execute_agent_node(Mock) → 停在 waiting，等用户审批
   └── report_agent_node        → pending
   │
   返回 {steps: [...6个节点的详细输出...], anomalies: [...]}
   │
   ▼
8. React → agentSteps 状态更新 → Trace Board 逐条动画展示
   │ 每张卡片显示多行思考过程，超过3行可展开
   │
   ▼
9. 用户查看审批卡片 → 点击「确认执行」
   │
   ▼
10. React → execute running(◉) → POST /api/campaigns/optimize
    │
    ▼
11. FastAPI → 实际修改出价/预算 → 持久化到 JSON
    返回 {changes: ["T001: bid 30→27", ...]}
    │
    ▼
12. React → execute done(✓) → report done(✓)
    │ 富文本展示调整摘要 + 节省预估
    │
    ▼
13. Dashboard 重新加载数据 → 显示新的 ROI 值
```

### 7.3 为什么选择这些技术？

| 技术 | 为什么选它 |
|------|-----------|
| **React** | 生态最大、组件化思维清晰、适合复杂交互 |
| **TypeScript** | 大型项目中类型安全减少 bug，编辑器智能提示 |
| **Tailwind CSS** | 不需要维护 CSS 文件，快速迭代 UI |
| **shadcn/ui** | 组件源码可控，不引入黑盒依赖 |
| **FastAPI** | Python 生态 + 性能接近 Node.js + 自动文档 |
| **Pydantic** | 自动参数校验，减少手写 if 判断 |
| **DeepSeek** | 国产高性价比，API 兼容 OpenAI |
| **LangGraph** | Agent 编排可视化，支持条件/中断/状态 |
| **JSON 文件** | Demo 阶段零配置，快速验证 |
| **Redis** | 高性能缓存 + 消息队列（Celery 后端）|
| **Chroma** | 轻量向量数据库，无需独立部署 |
