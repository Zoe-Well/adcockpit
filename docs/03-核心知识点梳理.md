## 一、LangGraph Agent 编排

### 1.1 StateGraph 核心概念

```python
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver

# 1. 定义状态 Schema
class AgentState(TypedDict):
    messages: List[Dict]
    data: Dict
    scene: str

# 2. 构建图
builder = StateGraph(AgentState)
builder.add_node("node_a", node_a_func)
builder.add_node("node_b", node_b_func)

# 3. 添加边
builder.add_edge(START, "node_a")                    # 普通边
builder.add_conditional_edges("node_a", router, {    # 条件边
    "path_1": "node_b",
    "path_2": "node_c",
})

# 4. 编译（带记忆）
app = builder.compile(checkpointer=MemorySaver())
```

**知识点**：
- `StateGraph` 是有向图，节点是处理函数，边是状态流转
- `TypedDict` 定义共享状态，每个节点返回部分状态更新
- `add_conditional_edges` 实现动态路由
- `MemorySaver` 提供 checkpoint 能力，支持中断恢复

### 1.2 Human-in-the-loop (HITL)，类似线程/进程中断去理解，重点就是触发中断，保持状态，然后从中断点返回继续执行

```python
from langgraph.types import interrupt, Command

def execute_node(state):
    if needs_approval(state["actions"]):
        # 暂停图，等待外部信号
        interrupt(state["pending_actions"])
    # 收到 Command(resume=...) 后从此处继续
    results = execute_actions(state["actions"])
    return {"execution_results": results}

# 恢复执行
graph.invoke(Command(resume={"approved": True}), config)
```

**关键点**：
- `interrupt()` 暂停图执行但不丢失状态
- `Command(resume=...)` 从暂停点恢复
- checkpointer 持久化暂停位置
- 适合审批流、人机协同场景

### 1.3 并行执行

LangGraph 不直接并行节点，但在节点内部可以并行：

```python
def data_agent(state):
    with concurrent.futures.ThreadPoolExecutor() as executor:
        futures = [executor.submit(fetch, platform) for platform in platforms]
        results = [f.result() for f in futures]
    return {"platform_data": results}
```

AdCockpit 实现：Data Agent 用 `ThreadPoolExecutor` 并行调用多平台 API。

## 二、Function Calling / Tool Use

### 2.1 LangChain @tool 装饰器

```python
from langchain.tools import tool

@tool
def get_top_campaigns(platform: str, days: int, metric: str, top_n: int) -> list:
    """获取指定平台按指标排名的 Top N 广告计划。

    Args:
        platform: 平台名称 ("douyin" 或 "tencent")
        days: 回溯天数
        metric: 排序指标 ("cost" 或 "roi")
        top_n: 返回条数
    """
    ...

# 工具定义会被自动转换为 OpenAI Function Calling Schema
```

**知识点**：
- `@tool` 装饰器自动从函数签名和 docstring 生成 Schema
- LLM 根据 Schema 决定何时调用、传入什么参数
- 工具返回值自动注入对话上下文

### 2.2 Function Calling 流程

```
用户输入 → LLM 分析 → 是否需要工具?
  ├─ 是 → 输出 function_call {name, arguments}
  │       → 框架执行实际函数 → 返回结果注入上下文
  │       → LLM 基于结果继续推理
  └─ 否 → 直接输出文本回复
```

### 2.3 Pydantic args_schema

```python
from pydantic import BaseModel, Field

class UpdateBidArgs(BaseModel):
    platform: str = Field(description="平台: douyin 或 tencent")
    campaign_id: str = Field(description="计划 ID")

tool = StructuredTool.from_function(
    func=update_bid,
    name="update_bid",

```

## 四、RAG (Retrieval-Augmented Generation)

### 4.1 基本概念

RAG = 检索 + 生成，解决 LLM 的"幻觉"和知识时效性问题。

```
用户提问 → Embedding 检索相关文档 → 拼接 Prompt → LLM 生成回答
```

### 4.2 在 AdCockpit 中的潜在应用

| 场景 | RAG 应用 |
|------|---------|
| 投放策略 | 检索历史优化记录，推荐相似案例的策略 |
| 内容生产 | 检索爆款素材库，匹配模板 |
| 故障诊断 | 检索历史故障案例，匹配根因和恢复方案 |
| 客户报告 | 检索历史 PPT 提纲，复用分析维度 |

## 五、记忆管理

### 5.1 记忆类型

| 类型 | 存储内容 | 生命周期 | 实现 |
|------|---------|---------|------|
| **短期记忆** | 当前对话上下文 | 单次会话 | `conversation_history` 列表 |
| **长期记忆** | 用户偏好、模板 | 跨会话 | `save/load_user_preference` |
| **工作记忆** | AgentState | 图执行期间 | LangGraph checkpointer |

### 5.2 记忆在 Agent 中的应用

```
短期记忆 → 多轮对话：记住用户上一轮说的平台和时间范围
长期记忆 → 个性化：记住用户默认的 ROI 阈值、关注哪些平台
工作记忆 → 中断恢复：审批中断后从断点恢复，不丢失执行进度
```

## 六、FastAPI + SSE 实时推送

### 6.1 SSE vs WebSocket

| 特性 | SSE | WebSocket |
|------|-----|-----------|
| 方向 | 单向（服务端→客户端） | 双向 |
| 协议 | HTTP | 独立协议 |
| 重连 | 浏览器自动 | 需手动实现 |
| 复杂度 | 低 | 中 |
| 适用场景 | 状态推送、日志流 | 聊天、实时协作 |

AdCockpit 选择 SSE：Agent 状态变更只需服务端推送给前端，客户端通过 REST 发送指令。

### 6.2 SSE 实现


### 7.4 Streamlit 限制与解决

| 限制 | 解决方案 |
|------|---------|
| 每次交互全量重跑 | 用 `session_state` 缓存数据，避免重复计算 |
| 无原生 WebSocket | SSE + `st.rerun()` 轮询模式，或后台线程读取事件 |
| CSS 隔离 | 注入全局 CSS 覆盖默认样式，`unsafe_allow_html=True` |

## 八、数字营销核心指标

### 8.1 广告投放指标

| 指标 | 公式 | 健康范围 | 说明 |
|------|------|---------|------|
| **ROI** | 收入/消耗 | > 2.0 | 投入产出比，< 2.0 通常需要优化 |
| **CPA** | 消耗/转化数 | < ¥50 | 单次转化成本，越低越好 |
| **CTR** | 点击数/展示数 | > 2% | 点击率，衡量素材吸引力 |
| **CVR** | 转化数/点击数 | > 2% | 转化率，衡量落地页/商品吸引力 |
| **消耗** | 实际花费 | — | 每日广告花费 |

### 8.2 优化策略

| 问题 | 策略 |
|------|------|
| ROI 偏低 | 降价、降预算、优化定向、换素材 |
| CPA 偏高 | 优化落地页、调整出价策略 |
| CTR 偏低 | 换素材、优化前 3 秒钩子 |
| CVR 偏低 | 优化商品详情页、发优惠券 |

## 九、设计模式与最佳实践

### 9.1 Agent 设计原则

1. **单一职责** — 每个 Agent 只做一件事
2. **无状态节点** — Agent 节点函数接收 State → 返回部分 State 更新
3. **可观测性** — 每个步骤的输入输出都可追踪
4. **优雅降级** — 工具调用失败不中断整个流程
5. **HITL 最小化** — 低风险操作自动执行，高风险操作才需确认

### 9.2 Mock 设计原则

1. **真实 Schema** — 与真实 API 结构一致
2. **异常覆盖** — 模拟超时、拒绝、限流等异常
3. **数据一致性** — 同一计划在不同查询中返回一致值
4. **易于替换** — 接口签名不变，只需切换实现

### 9.3 AI Coding 协作原则

1. **Spec 先行** — AI 写代码前先明确需求和架构
2. **分而治之** — 大任务拆成小文件，每次对话聚焦一个模块
3. **宪法约束** — 全局规则写死在 constitution.md 中
4. **原型参照** — 先做 HTML 原型作为 UI 标准，再让 AI 对标准实现
